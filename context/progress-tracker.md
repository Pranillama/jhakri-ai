## Architecture Decisions

- Liveblocks auth uses access tokens (`prepareSession`), not ID tokens. The route already performs the authoritative access check via `getAccessibleProject`, so it grants `FULL_ACCESS` to the verified room directly in the session token rather than relying on per-user room permissions. The room is still created up front (`getOrCreateRoom` with `defaultAccesses: []`, private) so it exists before connection and can later carry metadata; with access tokens the private default doesn't block the granted session. The spec's "return a session token" wording matches this model.
- Project ID == Liveblocks room ID. The create flow generates `<slug>-<suffix>` client-side and sends it as the project `id`; `POST /api/projects` persists it (validated, URL-safe) instead of always using the cuid default, so the DB record and the future room share one identifier. This extends 06's "use the schema's ID strategy" — cuid remains the fallback when no valid id is supplied. `POST` returns 409 on a `P2002` id collision (duplicated/retried create).
- Workspace route param is `[roomId]`, per spec 08. Because project ID == Liveblocks room ID, the dynamic segment value is the project ID; the create flow's `router.push(/editor/${project.id})` lands on it directly. Access is gated by a single `findFirst` (owner OR collaborator-by-email) in `lib/project-access.ts` — both "project missing" and "no access" collapse to `AccessDenied`, so the route never reveals which projects exist to unauthorized users.
- Collaborator emails are stored lowercased (normalized in `parseCollaboratorEmail` on invite) and membership is matched against the signed-in user's full set of **verified** emails (lowercased; primary always included), not just the primary — so an invite to a secondary address still grants access. Unverified emails are excluded so an unverified address can't be used to claim an invite. No local user table — Clerk is the source of truth for profile data, fetched on demand via the Backend API and never persisted (per spec 09). Clerk enrichment is best-effort: a failed lookup falls back to email-only rather than failing the request.
- Canvas state lives in Liveblocks Storage, not the DB. `useLiveblocksFlow` (suspense) is the single source of truth for nodes/edges in-session; the page stays a server component for access control while the canvas subtree is client-only (`LiveblocksProvider`/`RoomProvider` under `editor-room-shell`). The room is the project ID, so the `authEndpoint` reuses the same `getAccessibleProject` gate. Suspense + `ClientSideSuspense` keeps the static workspace chrome rendered while only the canvas waits on the realtime connection; a class-based error boundary isolates connection failures to the canvas instead of taking down the shell. Persisting the graph to Vercel Blob (`canvasJsonPath`) is deferred to a later unit per the workflow's "don't combine realtime canvas state and DB persistence" split rule.
- Ownership enforcement is atomic with the write (code-review fix). `PATCH`/`DELETE` no longer pre-check ownership in a separate query (TOCTOU window); they scope the mutation to `where: { id, ownerId }` and catch Prisma `P2025`. On that not-found path, `resolveMutationFailureStatus` in `lib/projects.ts` does a single existence read to pick 403 (exists, other owner) vs 404 (missing). The old `checkProjectOwnership`/`ProjectOwnershipResult` were removed.
