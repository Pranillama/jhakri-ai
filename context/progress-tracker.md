## Architecture Decisions

- Liveblocks auth uses access tokens (`prepareSession`), not ID tokens. The route already performs the authoritative access check via `getAccessibleProject`, so it grants `FULL_ACCESS` to the verified room directly in the session token rather than relying on per-user room permissions. The room is still created up front (`getOrCreateRoom` with `defaultAccesses: []`, private) so it exists before connection and can later carry metadata; with access tokens the private default doesn't block the granted session. The spec's "return a session token" wording matches this model.
- Project ID == Liveblocks room ID. The create flow generates `<slug>-<suffix>` client-side and sends it as the project `id`; `POST /api/projects` persists it (validated, URL-safe) instead of always using the cuid default, so the DB record and the future room share one identifier. This extends 06's "use the schema's ID strategy" â€” cuid remains the fallback when no valid id is supplied. `POST` returns 409 on a `P2002` id collision (duplicated/retried create).
- Workspace route param is `[roomId]`, per spec 08. Because project ID == Liveblocks room ID, the dynamic segment value is the project ID; the create flow's `router.push(/editor/${project.id})` lands on it directly. Access is gated by a single `findFirst` (owner OR collaborator-by-email) in `lib/project-access.ts` â€” both "project missing" and "no access" collapse to `AccessDenied`, so the route never reveals which projects exist to unauthorized users.
- Collaborator emails are stored lowercased (normalized in `parseCollaboratorEmail` on invite) and membership is matched against the signed-in user's full set of **verified** emails (lowercased; primary always included), not just the primary â€” so an invite to a secondary address still grants access. Unverified emails are excluded so an unverified address can't be used to claim an invite. No local user table â€” Clerk is the source of truth for profile data, fetched on demand via the Backend API and never persisted (per spec 09). Clerk enrichment is best-effort: a failed lookup falls back to email-only rather than failing the request.
- Canvas state lives in Liveblocks Storage, not the DB. `useLiveblocksFlow` (suspense) is the single source of truth for nodes/edges in-session; the page stays a server component for access control while the canvas subtree is client-only (`LiveblocksProvider`/`RoomProvider` under `editor-room-shell`). The room is the project ID, so the `authEndpoint` reuses the same `getAccessibleProject` gate. Suspense + `ClientSideSuspense` keeps the static workspace chrome rendered while only the canvas waits on the realtime connection; a class-based error boundary isolates connection failures to the canvas instead of taking down the shell. Persisting the graph to Vercel Blob (`canvasJsonPath`) was deferred to a later unit per the workflow's "don't combine realtime canvas state and DB persistence" split rule â€” done in 21-canvas-autosave, which layers a durable snapshot on top without changing this in-session source of truth.
- Starter template import replaces the canvas through React Flow's controlled-flow diff, not a bespoke clear path. `reactFlow.setNodes(next)`/`setEdges(next)` in this no-`defaultNodes` setup run xyflow's `getElementsDiffChanges` (verified in `@xyflow/react`), which emits `remove` changes for every current element absent from `next` plus `add`/`replace` for the new ones, and dispatches them through the same `onNodesChange`/`onEdgesChange` the Liveblocks integration owns. So passing the template's nodes/edges wholesale clears the old graph and adds the new one in one collaborative, synced operation â€” no separate "delete all" call, and no risk of stacking on top. Templates are deep-`structuredClone`d at import time because they are shared module constants and React Flow mutates node internals in place. The modal lives inside `<Canvas>` (not the shell) so the import handler has React Flow + Liveblocks context for `setNodes`/`setEdges`/`fitView`; only its open/close boolean is lifted to `editor-room-shell` and threaded down so the navbar (outside the room) can trigger it.

- Ownership enforcement is atomic with the write (code-review fix). `PATCH`/`DELETE` no longer pre-check ownership in a separate query (TOCTOU window); they scope the mutation to `where: { id, ownerId }` and catch Prisma `P2025`. On that not-found path, `resolveMutationFailureStatus` in `lib/projects.ts` does a single existence read to pick 403 (exists, other owner) vs 404 (missing). The old `checkProjectOwnership`/`ProjectOwnershipResult` were removed.
- Canvas snapshots are saved as **private** Vercel Blobs, not public ones. The blob path is deterministic (`canvas/{projectId}.json`), so a public blob would be a guessable URL exposing a project's architecture diagram to anyone who has it, with no auth check. Private access means the save/load routes are the only path to the content â€” both already gate on `getAccessibleProject` â€” at the cost of reading via the SDK's `get()` (server-side, token-authenticated) instead of a bare `fetch(url)`. Save/load are open to any project member (owner or collaborator), unlike the owner-only collaborator-management mutations, because the canvas itself is already collaboratively editable by anyone with room access â€” restricting the save endpoint further would just mean collaborators' edits silently stop persisting.
